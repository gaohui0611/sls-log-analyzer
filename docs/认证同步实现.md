# SLS 认证全自动同步 —— 技术实现文档

> 对应代码：`src/services/autoAuthSync.js` + `src/routes/auth.js` + `public/app.js`
> 端点：`POST /api/auto-sync-auth`

## 1. 要解决的问题

阿里云 SLS 日志查询接口（`getLogs.json`）需要三类凭证：

| 凭证 | 作用 | 存储位置 |
|------|------|---------|
| Cookies（含 HttpOnly） | 会话身份 | Chrome **磁盘** SQLite |
| `x-csrf-token`（CSRF Token） | 防跨站请求伪造 | 控制台页面**运行时内存** |
| `b3` header | 分布式链路追踪 ID | 控制台页面**运行时内存** |

**核心矛盾**：Cookies 在磁盘上，但 CSRF Token / b3 不在磁盘上 —— 它们只存在于已加载的 SLS 控制台页面的内存中（JS 变量 `window.__CONTEXT__` 或随请求 header 发出）。浏览器从不把它们持久化到文件。

早期方案的痛点：

| 方案 | 问题 |
|------|------|
| Puppeteer `launch()` 新开浏览器 | 需要用户在弹出的窗口**重新登录**阿里云 |
| Puppeteer `connect()` 连接现有 Chrome | 需用户手动以 `--remote-debugging-port` 启动 Chrome |
| 书签同步 | 用户要在 SLS 页面手动点书签，且 JS 读不到 HttpOnly Cookie |
| cURL 同步 | 用户要手动开 DevTools、复制请求 |

**目标**：用户只点一次「一键同步」，后端全自动完成，零手动操作。

## 2. 最终方案架构

```
点「一键同步」
   │
   ├─① 读 Cookies（磁盘解密）
   │     Chrome Cookies SQLite → 复制 → sqlite3 查询 → AES 解密
   │     产出：33 个阿里云 cookies（含 HttpOnly）
   │
   ├─② 抓 CSRF Token / b3（运行时拦截）
   │     Puppeteer 启动无头浏览器
   │     → 把①的 cookies 注入进去（自动登录态）
   │     → 导航到 sls.console.aliyun.com
   │     → 拦截发出的请求，从 header 抓 x-csrf-token / b3
   │     产出：csrfToken + b3
   │
   └─③ 合并存盘 + 实测验证
         写入 config.json，实际查一次 SLS 日志确认有效
```

**关键洞察**：cookies 能从磁盘读，token 必须从"跑起来的浏览器"读。那就用读到的 cookies 把一个无头浏览器"喂"成登录态，让它替我们去页面抓 token —— 两者串联，全自动化。

## 3. 阶段①：从磁盘解密 Chrome Cookies

### 3.1 定位文件

```
~/Library/Application Support/Google/Chrome/Default/Cookies       （旧版）
~/Library/Application Support/Google/Chrome/Default/Network/Cookies（新版）
```

代码扫描 `Default` + 所有 `Profile*` 目录，两者路径都试（见 `candidateCookiePaths()`）。

### 3.2 绕过文件锁

Chrome 运行时会锁定 Cookies 数据库。**复制一份到临时目录再读**，读完删除：

```js
const tmpDb = path.join(os.tmpdir(), 'sls-analyzer-cookies-' + Date.now(), 'Cookies');
await copyFile(dbPath, tmpDb);
```

### 3.3 读取（零依赖）

**不用 `better-sqlite3` 这类原生模块**，直接用系统自带的 `sqlite3` CLI，避免编译麻烦：

```bash
sqlite3 -separator $'\x1f' tmp.db \
  "SELECT name, host_key, hex(encrypted_value), path, is_secure, is_httponly
   FROM cookies WHERE host_key LIKE '%aliyun.com';"
```

分隔符用 `\x1f`（单元分隔符）而非 `\t`，防止 cookie 值里含制表符导致字段错位。

### 3.4 解密（最关键，也是踩坑最多的地方）

Chrome 在 macOS 上用 **AES-128-CBC** 加密 cookie 值，密钥派生过程：

```
Keychain 密码（"Chrome Safe Storage" 项）
     ↓ security find-generic-password -w -s "Chrome Safe Storage" -a "Chrome"
     ↓ PBKDF2-SHA1(password, salt='saltysalt', iterations=1003, dklen=16)
   AES-128 密钥
```

**加密格式（v10，新版本 Chrome）**：

```
[ "v10" 3字节前缀 ][ 32字节头部 ][ AES-CBC 密文 ]
```

解密代码：

```js
function decryptCookieValue(enc, key) {
    if (enc.slice(0, 3).toString('latin1') !== 'v10') return null;
    const decipher = crypto.createDecipheriv('aes-128-cbc', key, Buffer.alloc(16, 0x20)); // IV=16个0x20
    const dec = Buffer.concat([decipher.update(enc.slice(3)), decipher.final()]);
    return dec.slice(32).toString('utf8');  // 丢弃前 32 字节头部
}
```

**这里踩了两个坑（调试时排查的关键）：**

1. **切片顺序**。正确做法是「先对 `enc[3:]` 整段做 CBC 解密，再丢弃前 32 字节」。如果「先切掉头部再解密」，CBC 链式特性会使真实密文的首块错用 IV（应该用前一个密文块），解出乱码。
   - ❌ 错：`decipher(enc.slice(3 + 32))` → 乱码
   - ✅ 对：`decipher(enc.slice(3)).slice(32)` → 明文

2. **32 字节头部**。旧版 Chrome v10 格式是 `3字节前缀 + 密文`，新版变成了 `3字节前缀 + 32字节头部 + 密文`。诊断方法：用 Python（pycryptodome）端到端解密同一条 cookie 对比，发现丢弃前 32 字节后才是干净明文。

> v20 格式（AES-GCM）也做了支持：`3字节前缀 + 12字节nonce + 密文 + 16字节tag`。

## 4. 阶段②：抓取运行时 Token

Cookies 读到了，但 SLS 控制台发的请求带的 `x-csrf-token` 不在任何磁盘文件里。用 Puppeteer 解决：

```js
async function fetchRuntimeTokens(cookieMap) {
    const browser = await puppeteer.launch({ headless: 'new', ... });

    // 把阶段①解密出的 cookies 注入浏览器 —— 自动就是登录态，无需重新登录
    const cookies = Object.entries(cookieMap).map(([name, value]) => ({
        name, value: String(value),
        domain: '.aliyun.com', path: '/',
    }));
    await page.setCookie(...cookies);

    // 拦截请求，从 header 抓 token
    page.on('request', (req) => {
        const h = req.headers();
        if (h['x-csrf-token']) csrfToken = h['x-csrf-token'];
        if (h['b3']) b3 = h['b3'];
    });

    await page.goto('https://sls.console.aliyun.com', { waitUntil: 'networkidle2' });
    await new Promise(r => setTimeout(r, 8000)); // 等页面发出带 token 的请求
    await browser.close();
    return { csrfToken, b3 };
}
```

**为什么这样能拿到 token**：注入 cookies 后访问 SLS 控制台，因为有登录态，页面正常加载并初始化。控制台前端会自动发各类 API 请求（列项目、查日志等），每个请求 header 都带 `x-csrf-token`。Puppeteer 拦截这些请求即可拿到内存中的 token。

**无头模式（`headless: 'new'`）**：不弹窗、不抢占用户正在用的 Chrome、用独立临时 profile，所以用户零感知。

## 5. 阶段③：合并存盘 + 验证

```js
config.slsConfig = {
    cookies: { ...authData.cookies, timestamp: Date.now() },
    csrfToken: tokens.csrfToken,
    b3: tokens.b3,
    region,
};
await saveConfig(config);
```

随后 `/api/auth-status` 会用这套配置实际查一条 SLS 日志，确认认证有效（`isValid: true`）。

## 6. 为什么这套设计成立

| 设计选择 | 理由 |
|---------|------|
| 读磁盘 cookies 而非 Puppeteer 抓 | 拿得到 **HttpOnly** cookie（JS 读不到）；稳定、快 |
| Puppeteer 注入 cookies 抓 token | token 不在磁盘，只能靠"跑起来的浏览器"；注入 cookies 后**免重新登录** |
| 用 `sqlite3` CLI 而非原生模块 | **零新增依赖**，避免编译问题 |
| 复制数据库再读 | 绕过 Chrome 运行时的文件锁 |
| 无头独立 profile | 不干扰用户正在用的 Chrome，无需退出 |
| Keychain 首次授权 | macOS 安全机制，首次弹一次对话框点「始终允许」后永久记住 |

## 7. 边界与限制

- **仅 macOS**。Windows 用 DPAPI、Linux 用 PBKDF2+明文，解密逻辑不同（需另写）。
- **首次弹钥匙串授权**。macOS 安全机制，不可避免，但只一次。
- **Chrome 必须登录过 SLS**。否则 cookies 里没有有效会话。
- **Cookie/token 会过期**（约 30 天）。过期后重新点一次「一键同步」即可自动续期。
- **若 token 抓取失败**：通常是 cookies 已失效。表现为 SLS 页面被重定向到登录页，拦截不到带 token 的请求。解法：在 Chrome 里重新登录 SLS，再点一键同步。
